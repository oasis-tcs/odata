1      Introduction

This specification adds the notion of time-dependency to the Open Data Protocol (OData) without changing any of the base principles of OData. It defines semantics and a representation for temporal data, especially:

  • Semantics and operations for querying and modifying temporal data,
  • Vocabulary terms to annotate which data depends on time, and how.

2      Overview

When keeping track of time, the most important questions are:

        When did or will something happen?

        When did we learn that it happened?

This leads to two dimensions of time (in the meaning of point in time, measured as a date or date-plus-time value):

        application time, also called actual time, business time, effective time, or valid time, and

        system time, also called recording time, audit time, or transaction time.

Keeping track of time is typically done by storing each fact together with the period for which this fact is deemed valid or effective, using separate periods for each time dimension, and the time periods are part of the logical key for fact records. See [SQL:2011] or [Kulkarni] on how this is done in the SQL standard.

A consumer's perspective on these facts can be different: even if a time dimension is tracked internally, it may or may not be visible in a consumer's perspective, and even if visible it is often not considered part of the logical temporal object key. For example, an employee is still the same person even after switching to another department.

The goals of this extension are:

        Keep the models as simple as possible by allowing to hide time-dependency,

        Provide easy means for time-travel and time-series queries even if time is hidden, and

        Provide easy means for modifying time-dependent data.

More sophisticated queries are possible in models that make time visible.

2.1 Definitions

2.1.1 Application Time

Application time is used to describe facts that are known to change over time, e.g. the budget of a department, or which department an employee works for. This time dimension may capture planned changes in the future, e.g. transferring an employee to a new department next month or capturing next year's budget for a department. Both future and past facts can be changed.

2.1.2 System Time

System time is used to record when a fact has become known in the system of record. This time dimension does not extend into the future, and record entries are only added and are never changed.

System time is never manipulated directly. If it is visible, the corresponding properties are read-only.

2.1.3 Bitemporal

Bitemporal facts are tracked both in application time and system time to capture their change over time as well as the changes in the knowledge about these facts.

2.1.4 Time-Travel Query

Time-travel queries request the knowledge of facts as of a certain point in time. If the queried data is bitemporal, this requires specifying time coordinates for both time dimensions.

2.1.5 Time-Series Query

Time-series queries request how knowledge of facts has varied over a certain period of time. For bitemporal data this can be combined with time-travel in the second time dimension, or it can request all variations over a period in each dimension.

2.1.6 Temporal Object

A temporal object is a set of facts whose change over application and/or system time is tracked by the service.

2.1.7 Time Slice

A piece of temporal data with one or two time periods attached, documenting that the represented facts did not change during these time periods.

Time slices for the same temporal object are non-overlapping, so for any given point in (one- or two-dimensional) time there is at most one slice whose time periods contain that point in time.

Time slices for a temporal object need not cover the complete timeline or time plane. There can be points in time for which no time slice exists.

Time slices always use closed-open semantics, following [SQL:2011]. This means the start is part of the period, the end is not part of the period, and for directly adjacent time slices the end of the earlier time slice is identical to the start of the next time slice. The period start must be less than the period end.

Some software systems predating the availability of temporal databases and with date granularity for application-time use closed-closed semantics instead. Optional convenience functionality for this type of systems is described in chapter Application-Time with Closed-Closed Periods.

The time periods of a time slice may or may not be part of the OData entity key.

2.1.8 Time-Slice Entity Set

A time-slice entity set is an entity set in which each OData entity corresponds to an application-time slice and all application-time slices of a temporal object are part of the entity set.

For bitemporal facts system time is typically hidden and by default only data of time slices current in system time are shown.

2.1.9 Snapshot Entity Set

A snapshot entity set is an entity set in which each OData entity corresponds to a temporal object and by default only shows data of the current time slice.

2.2.2 Snapshot Entity Sets with Hidden Time-Dependency

2.2.2.1 Single Entities

For consumers only interested in the information as of now, the natural OData model uses the employee ID as the entity key and omits the application-time start and end dates:

Example 3: OData model with time-independent entity key and hidden application-time dimension

The OData service will expose only the information from the one record valid at the time the request is received and processed in the services time zone, i.e. the services notion of now or today.

Example 4: request processed before 2013-10-01

GET ~/Employees('E314')

results in

{

"@odata.context": "$metadata#Employees/$entity",

"ID": "E314",

"Name": "McDevitt",

"Jobtitle": "Junior"

}

Example 5: request processed at or after 2013-10-01

GET ~/Employees('E314')

results in

{

"@odata.context": "$metadata#Employees/$entity",

"ID": "E314",

"Name": "McDevitt",

"Jobtitle": "Senior"

}

Clients that are aware of the time-tracking capabilities underlying this seemingly timeless model may be interested in the information at a certain point in time. These time-traveling clients use the system query option $at to request data at a specific point in time:

Example 6: retrieve employee in the past

GET ~/Employees('E314')?$at=2012-01-01

has the same result as example 4.

Example 7: retrieve employee in the future

GET ~/Employees('E314')?$at=2025-01-01

has the same result as example 5.

The next type of question time-tracking aware clients will want to ask: how did the information change over time? The response has the same basic shape as the standard OData response to a single entity, i.e. a single object whose structure follows the declared entity types structure and contains the values of the last (i.e. newest) time slice. The period of each time slice is automatically included in the response. Time-period boundaries that are not declared properties of the requested entities are represented as instance annotations with terms from the vocabulary for temporal data.

If the newest time slice does not cover the requested period, additional time slices are nested as an instance annotation with term Temporal.History to fully cover the requested period. As these nested time slices have the same OData key as the newest (top-level) time slice, the OData key properties are omitted from the historic time slices.

There are no special values meaning beginning of time or end of time. Implementations can use sufficiently small/large values, e.g. the smallest/largest storable value of the underlying temporal data type. In the example below 9999-12-31 is used for end of time.

Example 8: retrieve employee over a period

GET ~/Employees('E314')?$from=2012-01-01&$to=2025-01-01

results in

{

"ID": "E314",

"Name": "McDevitt",

"Jobtitle": "Senior",

"@Temporal.From": "2014-01-01",

"@Temporal.To": "9999-12-31",

"@Temporal.History": [

{

"Name": "McDevitt",

"Jobtitle": "Junior",

"@Temporal.From": "2013-10-01",

"@Temporal.To": "2014-01-01"

},

{

"Name": "McDevitt",

"Jobtitle": "Junior",

"@Temporal.From": "2011-01-01",

"@Temporal.To": "2013-10-01"

}

]

}

2.2.2.2 Collections of Entities

Temporal query options also work with resource paths identifying collections of entities. Requesting data at a specific point in time results in the same basic shape as the corresponding request without the temporal query options, plus instance annotations for period boundaries if these are not declared properties:

Example 9: retrieve multiple employees at a past point in time

GET ~/Employees?$filter=contains(Name,'i')&$at=2012-01-01

results in one time slice for each employee matching the filter at the specified point in time

{

"@odata.context": "$metadata#Employees",

"value": [

{

"ID": "E314",

"Name": "McDevitt",

"Jobtitle": "Junior"

}

]

}

Requesting data for a period of time results in a collection of entities, each with an optional instance annotation with term Temporal.History:

GET ~/Employees?$filter=contains(Name,'i')&$from=2012-01-01&$to=2014-01-01

The interpretation of non-temporal system query options in combination with $at is straight-forward: the request is simply interpreted at that point in time.

The combination with $from and $to is less straight-forward and needs to be defined per standard system query option, e.g. a $filter is considered matching if the validity period of the time slice partially or fully overlaps with the request period and the filter condition is true for the property values of the time slice.

For details see Interaction with Standard System Query Options.

3      Vocabulary for Temporal Data

The following terms are defined in the vocabulary for temporal data [OData-VocTemporal].

The term TemporalSupport describes the temporal capabilities of an entity set. It can be applied to an entity set to describe this entity set and contained entity sets, or on a containment navigation property to describe just its entity set. It has a structured type with the following properties:

        ApplicationTime describes the granularity of the application-time dimension. Its value is a record of either type GranularityDate or GranularityDateTimeOffset.

o   Records of type GranularityDateTimeOffset MUST specify the property Precision.

o   Records of type GranularityDate MAY specify the property ClosedClosedPeriods, see chapter Application-Time with Closed-Closed Periods.

o   Records of both types MUST specify the property SupportedQueries, which is an enumeration value with the two flag values TimeTravel and TimeSeries.

        SystemTime describes the granularity of the system-time dimension. Its value is a record of type GranularityDateTimeOffset. It MUST specify the properties Precision and SupportedQueries.

        ObjectKey is a collection of property paths. The value combination of the referenced properties uniquely identifies the temporal object. If not specified, the temporal object key is identical to the OData entity key.

        SupportedActions is a collection of qualified names of temporal actions that may be bound to the annotated entity set.

At least ApplicationTime or SystemTime needs to be specified. If both are specified, the entity set is bitemporal.

The vocabulary defines four terms for the boundaries of the application- and system-time periods. These can be used as instance annotations, or as metadata annotations to mark declared properties as a period boundary. They use the abstract type Edm.PrimitiveType, but in fact their type MUST match the granularity of the corresponding time dimension:

        From is the start of the application-time period and included in the period.

        To is the end of the application-time period and excluded from the period.

        SystemFrom is the start of the system-time period and included in the period.

        SystemTo is the end of the system-time period and excluded from the period.

4      Temporal Requests

4.1 Temporal Expressions

A temporal expression is

        A date in the form dateValue, see [OData-ABNF]

        A timestamp in the form dateTimeOffsetValueInUrl, see [OData-ABNF]

        One of the literals min or max

        An arithmetic expression resulting in a date or timestamp value

The literals min and max are interpreted depending on the granularity of the time dimension. They represent the minimum and maximum values of the time dimension that the service will allow. Their values are service specific as they can depend on the capabilities of the underlying data store.

Note that there is no literal for today as clients and services can be in different time zones and thus can have different notions of which date is today. Clients are advised to use concrete temporal values and should avoid using the URL function now() with an application-time dimension of date granularity.

4.2 Querying Temporal Data

Temporal query options allow time-travel as well as time-series queries in both application time and system time. They take a temporal expression as their argument whose type MUST match the granularity of the corresponding time dimension.

Application-time query options can be combined with system-time query options when interacting with bitemporal data.

Temporal query options can be specified even if the directly addressed entity set does not track time. In this case the temporal query options dont have any effect other than being cascaded along $expand. [RH4] [RH5] 

Temporal query options can be specified within $expand. They override query options for the same time dimension specified on an outer nesting level, are evaluated on the expanded entities, and cascade to inner nesting levels unless overridden there.

Adding temporal query options to a request does not alter the basic response shape, it just adds the time boundaries for the addressed time dimensions to the represented OData entities if these are not already present as declared properties, and for time-series requests optionally a history of time slices for the represented OData entity.

Temporal query options can be used with data modification operations, in which case they do not alter the modification semantics of the request, they only affect the response shape, i.e. they only influence the implicit GET after the data modification.

4.2.1 Time-Travel Queries

4.2.1.1 Query Option $at

The $at query option takes a temporal expression as its argument.

It restricts the result to time slices whose application-time period contains the value of $at, i.e. with application-time start less than or equal to the value of $at, and application-time end greater than the value of $at.

For entity types with visible application time $at=<point-in-time> is essentially a shortcut for

$filter=@Temporal.From le <point-in-time>

and @Temporal.To gt <point-in-time>

The query option $at can be combined with $filter and $search, and in addition with either $systemat or $systemfrom and $systemto. Only entities satisfying all specified criteria are returned.

For entity types with hidden application time only those entities with a time slice satisfying all specified criteria are returned, and these entities are returned with the property values of that time slice.

4.2.2 Time-Series Queries

4.2.2.1 Query Options $from and $to

The query options $from and $to each take a temporal expression as their argument.

If $from is specified and $to is not specified, the value of $to defaults to max.

If $to is specified, $from MUST also be specified.

They restrict the result to time slices whose application-time period overlaps with the closed-open interval defined by the values of $from and $to, i.e. with application-time start less than the value of $to , and application-time end greater than the value of $from.

For entity types with visible application time $from=<start>&$to=<end> is essentially a shortcut for

$filter=@Temporal.From lt <end> and @Temporal.To gt <start>

The query options $from and $to can be combined with $filter and $search, and in addition with either $systemat or $systemfrom and $systemto. Only entities satisfying all specified criteria are returned.

If no $select is specified, each returned entity contains the application-time boundaries in addition to its default selection:

        declared properties of that entity type or one of its base types that is annotated with Temporal.From or Temporal.To defined in [OData-VocTemporal]

        instance annotations Temporal.From or Temporal.To if no declared property is correspondingly annotated, or in JSON responses with format parameter metadata=full

For entity types with hidden application time multiple time slices with the same OData key values may match the specified criteria, and a well-formed OData response can only contain one entity per combination of key values. In this case the response contains an OData entity with the values of the time slice with largest application-time start and an instance annotation with term Temporal.History containing the remaining time slices.

4.2.2.2 Query Options $systemfrom and $systemto

The query options $systemfrom and $systemto take a temporal expression as their argument.

If $systemfrom is specified and $systemto is not specified, the value of $systemto defaults to max.

If $systemto is specified and $systemfrom is not specified, the value of $systemfrom defaults to min.

They restrict the result to time slices whose system-time period overlaps with the closed-open interval defined by the values of $systemfrom and $systemto, i.e. with system-time start less than the value of $systemto, and system-time end greater than the value of $systemfrom.

The query options $systemfrom and $systemto can be combined with $filter and $search, and in addition with either $at or $from and $to. Only entities satisfying all specified criteria are returned.

If no $select is specified, each returned entity contains the system-time boundaries in addition to its default selection:

        declared properties of that entity type or one of its base types that is annotated with Temporal.SystemFrom or Temporal.SystemTo defined in [OData-VocTemporal]

        instance annotations Temporal.SystemFrom or Temporal.SystemTo if no declared property is correspondingly annotated, or in JSON responses with format parameter metadata=full

For entity types with hidden system time multiple time slices with the same OData key values may match the specified criteria, and a well-formed OData response can only contain one entity per combination of key values. In this case the response contains an OData entity with the values of the time slice with largest system-time start (and then application-time start for bitemporal models) and an instance annotation with term Temporal.History containing the remaining time slices.

4.2.3 Interaction with Standard System Query Options

If a time-travel query option ($at or $systemat) is specified or cascaded along $expand, all other system query options on the same expansion level are evaluated on the time slices whose period contains this point in time.

If $filter and $search are combined with temporal query options, only time slices satisfying all specified criteria are returned. The lambda operators any() and all() are not influenced by temporal query options, they are interpreted for each time slice on the filtered collection, meaning any related time slice satisfying the lambda expression and all related time slices satisfy the lambda expression. The lambda expressions however can contain sub-expressions working on time-slice boundaries.

The system query options $compute, $select, and $expand are evaluated per returned time slice.

The system query options $orderby, $top, $skip, and $count are evaluated on the OData entity level. Responses to time-series queries for hidden time periods may contain multiple time slices with the same OData key values, with the newest time slice (comparing first system-time start and then application-time start) being represented as a standard OData entity with an instance annotation Temporal.History containing the remaining time slices with the same OData key values. For these responses the property values of the newest (top-level) time slice are evaluated in $orderby, not considering the nested historic time slices. The response need not reflect the time-slice cut of the underlying data store, services may combine adjacent time slices with the same OData key values if they only differ in period boundaries. Services may also split time slices to align their boundaries with nested expanded time slices to represent a coarsest common slicing across an expand tree.

The system query option $apply defined in [OData-Aggregation] can be combined with time-travel query options ($at or $systemat) and are evaluated on the time slices whose period contains this point in time. The combination of $apply with the time-series query options ($from and $to) or ($systemfrom and $systemto) is not defined and is reserved for future versions of this specification.

4.2.4 Requesting Changes to Temporal Data

This specification does not define change-tracking combined with system-time query options.

Change tracking for time-slice entity sets without system-time query options works identical to non-temporal entity sets. If the entity set supports change-tracking combined with filtering on application-time period boundaries, the corresponding declared properties or the annotations @Temporal.From and @Temporal.To SHOULD be listed as FilterableProperties, see [OData-VocCap]. Clients can then use these properties in $filter or use the convenience shortcuts $at or $from and $to.

Change tracking for snapshot entity sets without temporal query options works identical to non-temporal entity sets and tracks changes as of now. This specification does not define change tracking for snapshot entity sets combined with temporal query options.

4.3 Modifying Temporal Data

System time is managed by the system and usually hidden in OData models. If visible in the model, system time slices may be directly manipulated via standard OData data modification operations, and this specification does not provide any additional means of manipulating system time slices.

This section and its subsections describe modifications in application time, both for

        snapshot entity sets with hidden application time, i.e. where each OData entity corresponds to a temporal object and by default only show data of the current time slice, and for

        time-slice entity sets with visible application time, i.e. where each OData entity corresponds to an application-time slice and all application-time slices of a temporal object are part of the entity set.

Modification operations fall into two categories:

        Direct manipulation of time slices, and

        Changes to a temporal object over a period of application time that affect multiple time slices without explicitly addressing each single affected time slice.

Snapshot entity sets do not allow direct manipulation of time slices.

For time-slice entity sets the first category is covered by standard OData data modification operations, with some specialties covered in the next section.

Changes to a temporal object over a period of application-time are covered in section Operations on Temporal Objects.

4.3.1 Operations on Time Slices

4.3.1.1 Create Time Slice

A POST request to a collection of entities creates a new entity in that collection, following the rules of [OData-Protocol] and the additional rules specified in this section.

For time-slice entity sets: instance annotations Temporal.From or Temporal.To can be used to specify the application-time period for the time slice to be created if the entity type of the new entity does not have modeled properties for application-time start or end.

If no application-time end is specified, the new time slice SHOULD end at the start of the next time slice for the same temporal object, or at the service-specific max if no next time slice exists.

If no application-time start is specified, the new time slice SHOULD start now. For snapshot entity sets with date granularity the service MUST make sure that a subsequent GET request without specifying $at will return the newly created time slice.

If the new time slice overlaps with existing time slices for the same temporal object, the service MUST either fail the request or avoid the overlap by adjusting the new or the existing time slices.

As a side-effect the service MAY join adjacent time slices that do not significantly differ.

For deep insert requests these rules apply also to nested entities.

Example 37: Enter next years budget for a department

POST ~/Departments('D08')/Budgets

Content-Type: application/json

 

{

"From": "2015-01-01",

"Budget": 1800

}

Returns (ID needed for next example)

{

"ID": "8d",

"From": "2015-01-01",

"To": "9999-12-31",

"Budget": 1800,

"ChangedAt": "2014-11-21T22:35Z "

}

4.3.1.2 Upsert Time Slice

A PATCH request to a URL that identifies a single entity is interpreted as an insert or an update, depending on whether the identified entity already exists.

For time-slice entity sets: the identified entity is a time slice. If the time slice does not exist, the request is treated as a Create Time Slice request. If the time slice exists, it is updated following the rules of [OData-Protocol]. In addition, if the request body contains the instance annotations Temporal.From or Temporal.To or corresponding declared non-key properties, the application-time period of the identified time slice is adjusted. If the adjusted time slice overlaps with existing time slices for the same temporal object, the service MUST either fail the request or avoid the overlap by adjusting the overlapping time slices.

For snapshot entity sets: the identified entity is a temporal object. If it does not have a current time slice, the request is treated as a Create Time Slice request. If it does have a current time slice, and the time slice starts before now, it is split into two time slices at now. The time slice starting now is then updated following the rules of [OData-Protocol]. For snapshot entity sets with date granularity the service MUST make sure that a subsequent GET request without specifying $at will return the updated time slice.

For deep update requests these rules apply also to nested entities.

Example 38: Change a department's budget and adjust period end

PATCH ~/DepartmentBudgets('8d')

Content-Type: application/json

 

{

"To": "2016-01-01",

"Budget": 1600

}

4.3.1.3 Update Time Slices overlapping with a Period

A PATCH request can be combined with a /$filter path segment to correct time slices that overlap with a period of application time without changing the period:

Example 39: Change a department's budget and adjust period end

PATCH ~/DepartmentBudgets/$filter(@foo)/$each?@foo=Department/ID eq 'D08' and @Temporal.To gt 2013-07-01 and @Temporal.From lt 2014-07-01

Content-Type: application/json

 

{

"Budget": 1400

}

4.3.1.4 Delete Time Slice

A DELETE request to a URL that identifies a single existing entity deletes the entity.

For time-slice entity sets: the identified time slice is deleted, either physically, or logically by hiding it from the service.

For snapshot entity sets: the identified entity is a temporal object with a current time slice. The application-time end of this time slice is set to now. For snapshot entity sets with date granularity the service MUST make sure that a subsequent GET request without specifying $at will not return the shortened time slice.

4.3.2 Operations on Temporal Objects

Changes to a temporal object over a period of application time can affect multiple time slices. For time-slice entity sets this can in principle be achieved by constructing batch requests with multiple operations on individual time slices. This not only puts burden on the client, it also makes it harder for services to check whether the request represents a consistent change of the temporal data.

Both can be avoided by providing convenience operations expressing the intent of the modification and leave the actual time slice manipulation to the service. Some of these operations can then also be used with snapshot entity sets that do not allow direct manipulation of time slices.

These convenience operations are modeled as bound actions defined in the vocabulary for temporal data [OData-VocTemporal]. Implementations SHOULD consider the preferences return=representation and return=minimal as specified in [OData-Protocol]. The convenience operations are atomic (all or nothing): they either succeed and produce the result described below, or they fail and do not change the temporal object.

Services may define specialized convenience operations for use cases not covered by this specification. Service-specific convenience operations SHOULD also be atomic.

4.3.2.1 Update during a Period

The UpdateObject action updates all time slices of a temporal object in a given period of application time after splitting partially overlapping time slices.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        From: the start of the period to update as a date or timestamp value matching the granularity of the application time dimension.

        To: the end of the period to update as a date or timestamp value matching the granularity of the application time dimension.

        Values: a structure matching a select-expand projection of the entity type of the binding parameter, containing the properties to update. It MUST NOT contain time slice boundaries and time slice keys.

This works identical to the SQL statement UPDATE FOR PORTION OF:

        All time slices of the identified temporal object whose application-time period fully or partially overlaps with the specified period to update are affected. Time slices that only partially overlap are split into two time slices, one not overlapping, and one fully overlapping.

        Then all fully overlapping time slices (including overlapping time slices created in the previous step) are updated following the rules for updating entities specified in [OData-Protocol].

        Gaps between time slices in the period to update are not affected.

On success it returns the collection of modified time slices.

4.3.2.2 Upsert during a Period

The UpsertObject action upserts a temporal object in a given period of application time after splitting partially overlapping time slices by updating existing time slices in that period and creating new time slices to close gaps in that period.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        From: the start of the period to upsert as a date or timestamp value matching the granularity of the application time dimension.

        To: the end of the period to upsert as a date or timestamp value matching the granularity of the application time dimension.

        Values: a structure matching a select-expand projection of the entity type of the binding parameter, containing the properties to upsert. It MUST NOT contain time slice boundaries and time slice keys.

This works similar to UpdateObject but closes gaps in the time period to update:

        All time slices of the identified temporal object whose application-time period fully or partially overlaps with the specified period to update are affected. Time slices that only partially overlap are split into two time slices, one not overlapping, and one fully overlapping.

        Then all fully overlapping time slices (including overlapping time slices created in the previous step) are updated following the rules for updating entities specified in [OData-Protocol].

        Gaps between time slices in the period to update are closed by creating new adjacent time slices. New time slices are created by copying the values of all properties except key properties and time slice boundaries from an adjacent preceding time slice if such a time slice exists, then the new time slice is updated following the rules for updating entities specified in [OData-Protocol]. If no preceding time slice exists, the time slice is created following the rules for creating entities specified in [OData-Protocol].

On success it returns the collection of modified time slices.

4.3.2.3 Update from a Point in Time, Stop on Change

The UpdateObjectFrom action updates properties of time slices of a temporal object from a given point in application time onwards and stops updating a property if its original value changes from one time slice to the next.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        From: the start of the period to update as a date or timestamp value matching the granularity of the application time dimension.

        To (optional): the end of the period to update as a date or timestamp value matching the granularity of the application time dimension.

        Values: a structure matching a select-expand projection of the entity type of the binding parameter, containing the properties to update. It MUST NOT contain time slice boundaries and time slice keys.

This works as follows:

        The time slice of the identified temporal object containing From is split if it does not start with From.

        The time slice starting with From is updated following the rules for updating entities specified in [OData-Protocol].

        For each subsequent time slice: all properties to update that have the same value as the original value in the preceding time slice are updated, i.e. updating of a property stops if its value in the next time slice differs from the original value in the current time slice.

        If To is specified, this stops at latest when a time slice is reached that contains To. If this time slice does not start with To, it is split and the time slice ending with To is updated following the update same value principle.

        Gaps between time slices in the period to update are not affected.

On success it returns the collection of modified time slices.

4.3.2.4 Synchronize Full Timeline

The SynchronizeObject action synchronizes time slices of a temporal object to match the provided full application timeline. This can be implemented as a full purge and replace, or by adjusting only time slices that differ in property values or application-time boundaries.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        Timeline: a collection of entities of the entity type of the binding parameter, containing the full application timeline of the temporal object to synchronize.

After applying this action all application-time slices of the identified temporal object have the same application-time boundaries and the same property values (excluding visible system-time boundaries) as the provided timeline. For bitemporal collections this is evaluated at the system time the action is processed.

 

On success it returns 204 No Content because the request already contains the desired result.

4.3.2.5 Delete during a Period

The Delete action deletes all time slices of a temporal object in a given period of application time after splitting partially overlapping time slices.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        From: the start of the period to delete as a date or timestamp value matching the granularity of the application time dimension.

        To: the end of the period to delete as a date or timestamp value matching the granularity of the application time dimension.

This works as follows:

        All time slices of the identified temporal object whose application-time period fully or partially overlaps with the specified period to delete are affected. Time slices that only partially overlap are split into two time slices, one not overlapping, and one fully overlapping.

        Then all fully overlapping time slices (including overlapping time slices created in the previous step) are deleted following the rules for deleting entities specified in [OData-Protocol].

On success it returns the collection of deleted time slices.

4.3.2.6 Delete from a Point in Time, Stop on Change

The DeleteObjectFrom action deletes time slices of a temporal object from a given point in application time onwards and stops if a property changes its value from one time slice to the next adjacent. It also stops if it reaches a gap.

It is bound to a collection of entities and its non-binding parameters are

        ObjectKey: a structure matching a select-expand projection of the entity type of the binding parameter, containing all and only properties of the temporal object key of that collection.

        From: the start of the period to delete as a date or timestamp value matching the granularity of the application time dimension.

        To (optional): the end of the period to delete as a date or timestamp value matching the granularity of the application time dimension.

This works as follows:

        The time slice of the identified temporal object containing From is split if it does not start with From.

        The time slice starting with From is deleted following the rules for deleting entities specified in [OData-Protocol].

        For each subsequent time slice: the time slice is deleted if each property except time period boundaries has the same value as in the preceding (just deleted) time slice. If there are any differences, the action stops.

        If To is specified, this stops at latest when a time slice is reached that contains To. If this time slice has the same property values and does not start with To, it is split and the time slice ending with To is deleted.

On success it returns the collection of deleted time slices.

4.3.3 ETags

Optimistic concurrency for temporal data requires that each time slice has its own ETag.

Time-slice entity sets need no special consideration as each time slice is an OData entity.

For snapshot entity sets time-travel and time-series queries can return time slices other than the current ones. It is important that these time slices are returned with ETags that differ from the ETags of the corresponding current time slices: only the current time slices are affected by data modification requests and using different ETags helps prevent unintentional data loss.

4.3.4 Read, Edit, Navigation, and Association URLs

Time-slice entity sets need no special consideration as each time slice is an OData entity.

For snapshot entity sets read, edit, navigation, and association URLs are not specific to time slices, they are specific to the time-independent OData entity.

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

5      Application-Time with Closed-Closed Periods

Some software systems predating the availability of temporal databases and using application-time granularity date use closed-closed semantics.

Temporal services on top of these systems can either convert their period end boundaries on-the-fly by adding one day on the way out and subtracting one day on the way in, or alternatively delegate that task to the client:

Example 41: application-time period with closed-closed semantics, client needs to add one day to the inclusive To to calculate the standard exclusive To.

<EntityType Name="EmployeeClosedClosed">

<Annotation Term="Temporal.From" Path="From" />

<Annotation Term="Temporal.To">

<Add>

<Path>ToInclusive</Path>

<Duration>P1D</Duration>

</Add

</Annotation>

</EntityType>

In addition, these services MAY support the temporal query option $toInclusive instead of $to. Using a different query option makes the significantly different semantics more obvious to client developers.

Example 42: retrieve employees over a period of application time, with inclusive period end

GET ~/EmployeesClosedClosed?$from=2012-01-01&$toInclusive=2024-12-31

Support for this alternative query option MUST be indicated by specifying application-time granularity date with the property ClosedClosedPeriods and a value of true.

Example 43: entity set with closed-closed application-time period supporting $toInclusive instead of $to

<EntitySet Name="EmployeesClosedClosed">

<Annotation Term="Temporal.TemporalSupport">

<Record>

<PropertyValue Property="ApplicationTime">

<Record Type="Temporal.GranularityDate">

<PropertyValue Property="ClosedClosedPeriods" Bool="true" />

</Record>

</PropertyValue>

</Record>

</Annotation>

</EntitySet>

6      Conformance

Conforming services MUST follow all rules of this specification for the time dimensions and temporal system query options they support. They MUST implement all time dimensions they advertise via the TemporalSupported annotation.

Conforming clients MUST be prepared to consume a model that uses any or all constructs defined in this specification and MUST ignore any temporal constructs not defined in this version of the specification.
